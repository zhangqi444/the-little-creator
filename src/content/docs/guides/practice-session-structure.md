A good practice session doesn't just happen. Without structure, kids default to working on what's interesting to them (usually the robot) while ignoring what most needs doing (usually the Innovation Project or Core Values prep). This guide gives you a repeatable framework for 60–120 minute sessions.

## The core structure (any length session)

Every session has four phases, scaled to your available time:

| Phase | 60 min | 90 min | 120 min |
|---|---|---|---|
| **Check-in** | 5 min | 5 min | 5 min |
| **Focused work** | 40 min | 60 min | 85 min |
| **Test / share** | 10 min | 18 min | 20 min |
| **Debrief** | 5 min | 7 min | 10 min |

---

## Phase 1 — Check-in (5 minutes)

**Purpose:** Settle the group, establish focus, surface any issues from last session.

**How:**
- Everyone gathers — no building yet
- One question, quick round: "What's one thing you want to accomplish today?"
- Coach announces the session's primary goal (one sentence): "Today we're getting missions 3 and 4 reliable. Everything else is secondary."
- If there's a time constraint (tournament in X weeks), name it

**What not to do:** Skip this. Sessions without a stated goal drift. Five minutes of alignment saves twenty minutes of chaos.

---

## Phase 2 — Focused work (bulk of the session)

**Purpose:** Make actual progress on one or two defined things.

### Split the team, if you have enough people

FLL teams of 4+ benefit from splitting into parallel workstreams rather than everyone crowded around the robot:

| Workstream | Who | What |
|---|---|---|
| Robot | 2–3 kids | Building, programming, testing specific missions |
| Innovation Project | 2–3 kids | Research, presentation prep, prototype building |
| Core Values | 1 coach + rotation | Brief team activity, or debrief from last session |

Rotate kids between workstreams across sessions — everyone should touch both the robot and the IP work.

### For robot work: use the "plan → build/code → test" loop

Don't let kids build or code without a plan. Before touching the robot:
1. **State the goal** (30 seconds): "We want the robot to complete mission 3 from starting position A in under 20 seconds."
2. **Identify the one change** you're making: "We're adjusting the gyro turn from 92° to 88° and testing."
3. **Test 3 times** (not once). One run proves nothing.
4. **Decide:** keep the change, revert, or try something different.

This loop takes 5–10 minutes per iteration. Focused teams run 4–6 iterations per session. Unfocused teams spend 40 minutes debating and run one test.

### For Innovation Project work: use the session to make one decision

IP work often stalls because the task is too open. Give the IP sub-team a specific decision to make by end of session:
- "Choose the top 3 problem candidates from our brainstorm — be ready to explain why."
- "Write the outreach email to [expert] and have it ready to send."
- "Agree on the solution structure and sketch it on paper."

One decision per session keeps IP work moving without overwhelming kids.

### For Core Values: short and embedded, not a separate activity

Core Values work is most effective when it's integrated into normal practice, not tacked on as a separate "values lesson." Options:
- Open with a 5-minute activity from [Core Values Activities](/guides/core-values-activities/) — then get to work
- Notice and name values as they show up: "That was gracious professionalism — you helped Jamie debug without being asked."
- Use the debrief (Phase 4) to reflect on how the team worked together, not just what they accomplished

---

## Phase 3 — Test / share (10–20 minutes)

**Purpose:** See progress, share across workstreams, celebrate small wins.

**How:**
- Robot sub-team does one demonstration run — not practice, a *demonstration* for the whole team
- IP sub-team shares what they decided or produced
- Everyone watches; coach facilitates brief reactions: "What did you notice?" "What's still unresolved?"

**Why this matters:** Kids working on the IP don't see the robot progress, and vice versa. Sharing creates team ownership of all workstreams — critical for the Core Values judging session, where every team member needs to speak about both.

---

## Phase 4 — Debrief (5–10 minutes)

**Purpose:** Close the loop, set up the next session, capture what you learned.

**Standard three questions:**
1. "What did we accomplish today?" (Concrete — what actually changed or progressed?)
2. "What's the one thing blocking us right now?"
3. "What's the goal for next session?"

Write the answer to question 3 somewhere — a sticky note on the robot, a shared notes doc, the whiteboard. Start next session by reading it.

**Optional fourth question** (especially after a frustrating session): "What was something that went better than expected?" Helps kids leave on a forward-looking note even when the session was hard.

---

## Session planning checklist (for the coach, before each session)

- [ ] What is the primary goal for today? (One sentence)
- [ ] Where are we in the season? (Adjust balance of robot / IP / Core Values accordingly)
- [ ] What's the current blocker? (Address it explicitly in focused work)
- [ ] What supplies or setup do I need to prepare in advance? (Mat, laptop, materials)
- [ ] How will I split the team today?

## Adjusting for season phase

| Season phase | Robot focus | IP focus | Core Values focus |
|---|---|---|---|
| **Early (Aug–Sep)** | Explore/build; no pressure to complete missions | Brainstorm + research | Team identity; introductory activities |
| **Mid (Oct)** | Mission-by-mission completion; start tracking success rates | Solution development + prototype | Reflection on how team is working |
| **Late (Nov)** | Consistency; lock run strategy | Presentation rehearsal | Judging prep; gracious professionalism |
| **Pre-tournament week** | One or two tune-up runs only; no major changes | Final presentation run-throughs | Tournament logistics + team mindset |

The biggest mistake coaches make in late season: spending 80% of practice on the robot because it's concrete and satisfying, while the IP and Core Values prep falls behind. Use the session structure to enforce balance.

## Handling common session problems

### "We only have 45 minutes"

Compress: 3 min check-in → 30 min focused work (pick ONE thing only) → 7 min debrief. Don't skip the debrief.

### "Kids are fighting about robot design"

Pause the session. Name it: "We're stuck on a disagreement. Let's spend 5 minutes deciding this properly." Use the Decision Jam activity from [Core Values Activities](/guides/core-values-activities/). Then move on.

### "Nobody wants to work on the Innovation Project"

Don't make it optional. Split the team explicitly: "You three are on the robot; you two are on the IP. We switch in 20 minutes." Giving IP work a distinct time and team makes it real.

### "We're running out of time before the tournament"

Triage ruthlessly: (1) robot must be reliable for your core missions; (2) IP must be presentable; (3) Core Values prep is one run-through of the judging format. Cut everything else.

### "Sessions keep running over time"

Set a visible timer in the room. When it goes off, stop. The debrief is not optional even when you're running over — 5 minutes of structured reflection is worth more than 5 minutes of additional unfocused building.
